The idea

Stop talking to one model. Run a team of them.

A normal AI coding session is a conversation with a single model. agglomeration-platform asks a different question: what if one orchestrator could run several different AI models at once, give each a job, and combine their answers? Different models have different strengths and blind spots — so a panel of them, cross-checking each other, is more reliable than any one alone.

A hub that orchestrates

Your Claude Code session becomes the hub — the conductor. It never does the grunt research or coding itself; it briefs, dispatches, verifies, and synthesizes.

Real tools, not API calls

Each worker is a genuine interactive model CLI running in its own terminal pane. You can watch any of them work, or attach and take over.

Agglomerate & cross-check

The platform agglomerates many agents onto a problem, then has them attack each other's conclusions — an adversarial pass that catches what a single model misses.

The name says it: it's a platform that agglomerates — clusters many independent AI agents into a coordinated whole. Everything is driven from one place with /ap:* commands, and every worker is something you can see, attach to, and inspect.

Vocabulary

Four words that explain the whole system.

Once these four terms click, every command below reads naturally.

Hub your session

The orchestrating Claude Code session running the /ap:* commands. It plans, dispatches work, verifies results, and writes the final artifacts. In messages it identifies itself as From: hub.

Agent a configured role

A model role defined in config, given a call-sign from the NATO phonetic alphabet — alpha, bravo, charlie … zulu. An agent is the identity; it isn't doing anything yet.

Worker an agent at work

A spawned agent actively working a task — a live model TUI in its own tmux pane. This is the thing you can attach to and watch type. Workers do all the retrieval and coding; the hub never does.

Cluster a color family

A color grouping of agents, shown as the pane-border color so you can tell crews apart at a glance: azure, sage, amber, slate, ivory, violet.

So a full sentence in this world reads: “the hub spawned agent bravo as a worker, with a sage pane border because bravo is in the sage cluster.”

The core workflow

Explore → Design → Implement.

The platform's main pipeline takes an idea from open question all the way to a merged pull request. Each stage hands a concrete artifact to the next.

1

Explore

/ap:explore surveys a hard topic from multiple angles — several workers research in parallel, then attack the synthesis as adversaries. Produces a landscape doc and a design hand-off.

2

Design

/ap:design turns a topic (or that hand-off) into a single buildable design doc — Problem, Goal, Architecture, Components, Testing, Success Criteria — cross-verified across models.

3

Implement

/ap:implement consumes that design doc: a worker plans, builds, and self-verifies on its own branch while the hub cross-verifies and runs a fix-loop, finishing with a PR.

Other entry points: /ap:quick — skip straight to a small change /ap:autoresearch — run real experiments /ap:review — learn from what happened

You don't have to run the whole pipeline. Need a quick, clearly specified change? Jump straight to /ap:quick. Want to empirically test ideas rather than just read about them? /ap:autoresearch runs an adaptive loop of real experiments. The pipeline is the backbone; the other commands are on-ramps and side-quests.

The commands

Every command, and when to reach for it.

All commands are namespaced /ap: and run from the hub session. They fall into three families: build & research, operations, and cross-repo.

Build & research

/ap:exploreresearch

Deep, multi-angle survey of a topic. 2–3 workers research in parallel; the hub synthesizes a landscape, runs a confidence gate, then dispatches the same workers as adversaries to challenge the synthesis before writing a final landscape doc + design hand-off.

When: you want to understand a space — SOTA approaches, architectures, a hard research question — without committing to a buildable plan yet.

# survey a landscape from several angles /ap:explore survey current approaches to retrieval-augmented agents
/ap:designbuild

Cross-verified multi-model investigation that produces one deploy-schema design doc (Problem / Goal / Architecture / Components / Testing / Success Criteria) which passes an audit gate. Runs a fast hub-only path for simple topics, or escalates to a 2–3 worker ensemble.

When: you know roughly what you want to build and need a rigorous, buildable spec — the exact artifact /ap:implement consumes.

# produce a buildable design doc /ap:design Design a token-bucket rate limiter for the API gateway
/ap:implementbuild

Consumes a design doc: audits it, spawns one worker to plan / implement / self-verify on its own feat/implement-<topic> branch, then the hub cross-verifies and runs a bounded fix-loop. Ends with a finish menu + teardown — push and open a PR. Single-repo.

When: you have a design doc (usually from /ap:design) and want it built and verified hands-off.

# build the design doc that design produced /ap:implement docs/specs/2026-06-23-rate-limiter-design.md
/ap:quickbuild

The light, autonomous path. One worker (default codex) implements a small, clearly specified single-repo change unattended on its own feat/quick-<topic> branch; the hub briefs, runs one implement turn, does a light verify, and finishes by default. No research, no design doc, no interactive gates.

When: the change is small and the spec is obvious — a flag, a helper, a focused fix — and you don't need the full pipeline.

# one worker, no ceremony /ap:quick add a --json flag to the status command
/ap:autoresearchresearch

An executable research session. You and the hub lock a measurable metric, sweep the SOTA, spawn 2–3 persistent codex workers, then adaptively dispatch single-config experiments until a target, plateau, or budget stop fires. Ships a full research-validity layer (verify by re-execution, sanity checks, coverage, lineage, an independent inspector). Explore-only — it never touches your real source.

When: you want to empirically test ideas and measure outcomes, not just survey the literature.

# lock a metric, then run experiments toward it /ap:autoresearch minimize val loss on a CIFAR-10 subset under a 5-min train budget

/ap:reviewresearch

The feedback loop. Every command writes forensics at teardown; /ap:review surveys what's new since you last ran it, shows how often each pattern has recurred over the life of the project, suggests one next action per cluster, then archives what it showed you so the next run starts clean. Zero arguments.

When: periodically — to spot recurring failures, flaky providers, or patterns worth turning into a fix.

# what went wrong (or right) lately? /ap:review

Operations

Health check — tmux, pane borders, state dir, config, and which providers are installed — plus an interactive picker to choose the active provider ensemble for /ap:design.

When: first-time setup, or to confirm the environment is sane.

/ap:listops

Show every active worker across topics — its pane and state — or scope to a single topic.

When: you want to see what's currently running.

Gracefully end workers — a DONE banner, then archive their state. Stop one, a topic, or everything with --all --yes.

When: a run is finished or stuck and you want a clean teardown.

Cross-repo

/ap:bridgecross-repo

When a task needs changes in a different git repository than the one you're in, bridge opens one persistent worker inside that other repo and co-develops with it over open-ended rounds — relaying real decisions back to you, answering trivia from context — and finishes as a PR there. All without your hub session leaving its current repo.

When: a change belongs in repo B, but you're working in repo A and don't want to context-switch away.

# co-develop in another repo, finish as a PR there /ap:bridge --repo /path/to/other-repo "add OAuth login to the gateway"

Deep dive · /ap:autoresearch

The research engine, in full.

Most commands move information around. /ap:autoresearch is different: it runs real experiments — code that executes, trains, and measures — and steers a fleet of workers toward a number you defined. Because the workers report their own scores, the whole system is built around one uncomfortable premise:

“A worker's self-reported metric is a claim, not evidence.”

Earlier versions trusted the number a worker printed. This one doesn't. Every reported result is re-executed, sanity-checked, and — for any new record — independently re-implemented by a different model before it is allowed to win. That verification layer is most of what makes autoresearch worth the machinery.

How it's designed

Four principles shape everything below.

Advisor-driven, metric-locked

You and the hub lock a measurable metric up front — direction, floor, target, how many corroborating runs count as done. Completion is then judged numerically, never by a worker saying “this looks good.” Even in hands-off mode the hub always stops to confirm the metric with you first.

Explore-only, with a promotion gate

Workers run in throwaway branch directories and never touch your real source. The only path to production code is an explicit hand-off into /ap:design → /ap:implement. Research and shipping are deliberately separate steps.

Idea quality ≠ execution quality

A bad number can mean a bad idea — or a buggy, leaky, under-trained run. The system separates the two, so a broken execution gets retried, not mistaken for a refuted idea. Only an execution-verified bad result retires an approach.

Trust nothing; verify mechanically

Six task-agnostic gates (below) replace honor-system trust with re-execution, numeric bounds, and independent reproduction. None of them needs to understand your specific problem to catch a faked, leaky, or over-fit result.

How it works — the session lifecycle

A run moves through four stages. You're involved at the start — locking the metric and the budget; after that it's an autonomous loop until a stop condition fires.

Stage 1 · Setup

Lock the target

The hub confirms a measurable metric with you (floor, target, corroboration count), sweeps current SOTA into a reference doc, and sizes the run: 2–3 workers + a time budget.

Stage 2 · Spawn

Launch the executors

2–3 persistent codex workers spawn once into their own panes. Each is handed a different approach family, so they explore in parallel rather than in lockstep.

Stage 3 · Adaptive loop

Experiment & steer

Workers run single-config experiments; the hub scores each result, verifies it, updates a leaderboard, and dispatches the next experiment — until a stop condition fires.

Stage 4 · Wind-down

Synthesize & hand off

The hub writes a landscape doc, preserves every experiment's code and logs, and emits a hand-off naming the winner — ready to promote via /ap:design.

Inside the adaptive loop

Each pass is mechanical. The hub never asks “should I keep going?” — stopping is owned by exactly two checkpoints (the budget gate and the completion check), and nothing else may halt it.

  1. Wait for a worker to finish an experiment (it writes one result.json).
  2. Budget gate — if the time budget is spent or you asked to halt, stop and finalize.
  3. Score the result into the leaderboard, then verify it: re-execute, sanity-check, and — for a new best — independently re-implement.
  4. Completion check — is the floor + target met with enough corroborating runs, or has the search plateaued across enough approaches? If so, stop.
  5. Dispatch the next experiment to any idle worker — either a Draft (open a new approach family) or an Improve (change exactly one variable on a named parent run, so the gain is attributable). Then loop.
Two move types: DRAFT — open a new, orthogonal approach family IMPROVE — change one variable on a parent run

The validity layer — six gates against a faked win

This is what separates autoresearch from “ask three models to optimize something.” Every reported result runs this gauntlet; only survivors can lead the board or trigger a stop. The first three ask “is this result real?”; the next two ask “are we exploring well?”; the last is the anti-gaming capstone.

A1

Verify by re-execution

Defends against: fabricated or un-reproducible numbers.

The hub re-runs the worker's own declared scoring command outside its pane and recomputes the metric. Input files are sha256-snapshotted at score time, so a worker that edits its inputs before verification is caught. Verdict: verified / mismatch / unavailable / pending.

A3

Sanity & integrity

Defends against: buggy / leaky / under-trained runs that still look valid.

Task-agnostic flags: too-good (beats a known ceiling), too-fast (ran quicker than real training could), log-contradiction (status “ok” but the logs show a crash or OOM), missing integrity attestation (never affirmed there was no train/test leakage), and knob-drift (quietly changed a locked hyperparameter).

A2

Infeasible vs refuted

Defends against: mistaking a broken run for a disproven idea.

A result that couldn't be validly executed (an A1 mismatch or a hard sanity flag) is quarantined into a separate non-ranked group — it can't win and can't trigger an early stop — and the idea is re-dispatched. A result that ran cleanly but simply lost is real evidence and steers the search normally.

B1

Coverage & diversity

Defends against: premature convergence on one approach.

Experiments are bucketed into approach families. The plateau-stop won't fire until enough distinct families have produced valid results and none is still improving — so the loop can't declare victory after merely over-tuning a single idea.

B2

Operators & lineage

Defends against: wins you can't attribute.

Every experiment is typed: a Draft opens a new family; an Improve changes exactly one knob versus a named parent. Lineage records what changed, so a metric gain traces to a single variable — and a leader that quietly moved several knobs at once is flagged [multi-change].

C1

Independent re-implementation · capstone

Defends against: reward-hacking baked into the worker's own scoring code.

When a worker sets a new record, a different model family (the Claude hub) rebuilds the experiment from the worker's run-card alone — never reading the worker's code — and re-derives the metric. A confident not-reproduced demotes it; an honest run that's merely hard to reproduce is marked inconclusive and never punished.

When does it stop?

Three conditions, and only three — the loop is otherwise unbounded by design.

Target reached

The floor and target metric are met and corroborated by the required number of independent at-target experiments.

Plateau

Enough distinct approach families have been explored and every one has stalled — there is no obvious next gain to chase.

Budget / halt

The time budget is exhausted, or you say “stop.” The hub finalizes whatever it has into a clean, archived result.

What it can achieve

  • A number you can actually trust. Every reported win has survived re-execution, mechanical sanity checks, and — if it's a record — independent re-implementation by a different model.
  • A defensible stopping point. The decision to stop is numeric and reproducible — floor + target + corroboration, an approach-aware plateau, or a budget — not a vibe.
  • A fully auditable trail. Every experiment's code, result, and logs are preserved; a landscape doc records what was tried, what won, and why the search stopped.
  • Honest negatives. Because broken executions are quarantined and retried, an idea is retired only when it genuinely lost a valid run — the search doesn't discard good ideas over bad plumbing.
  • A clean path to production. The winner is handed off as score-handoff.md → /ap:design → /ap:implement, the only route from experiment to real code.
Honest limits. Workers run with their sandbox disabled (--dangerously-bypass-approvals-and-sandbox) and network access on — isolation is honor-system, so run it on a scratch machine or worktree, never anywhere with secrets or production data. And “corroboration” today means several distinct at-target experiments, not the same config across many random seeds — so a genuinely lucky seed can still satisfy completion. A statistical multi-seed gate is designed but not yet shipped.

Under the hood

The result every worker writes▸

Each experiment ends with a single frozen-schema result.json: branch id, approach label, metric name + value, status (ok / fail / timeout / cost_blown), runtime, log paths, checkpoint, and notes — plus optional integrity attestations (no train/test overlap, target not in features, trained steps, seed) and a run-card (data spec + metric formula) that the C1 inspector rebuilds from. A result is only valid if its metric is non-null exactly when the status is ok.

The leaderboard, and how a winner is chosen▸

Results are scored into a ranked scoreboard.md. Valid ok rows get integer ranks (sorted by the metric's direction, with deterministic tie-breaks); infeasible runs are demoted to a visible-but-excluded x-rank group; failures sit below. Rank 1 is always a valid result — an invalid one can never be crowned, and the completion check reads only integer ranks, so a quarantined “win” can't trigger a stop.

Pure core, side-effecting shell▸

All the decision logic — metric parsing, scoring, the completion check, and the six validity gates — is written as pure functions with the filesystem injected, so it is unit-tested without spawning a single pane. The actual experiment dispatch, tmux nudges, and disk writes live only in the command layer. It's a faithful TypeScript port of the original research shell scripts, with the worker wire-format frozen for drop-in compatibility.

Worker liveness & recovery▸

A persistent monitor watches each worker's event stream and flags it stale or stuck if it goes quiet, so the hub can probe or abort a hung run. If the hub itself restarts, it reconstructs each worker's state from disk — a crashed run is reconciled from its last events, not lost. And a worker that produces three valid-but-far-from-leader results in a row is automatically retired so effort flows to the promising approaches.

Providers

Four interchangeable model back-ends.

A provider is the model CLI a worker runs. The set is deliberately closed — adding one is a config row plus a real-world test, not an open free-for-all — so every worker behaves predictably. Each provider runs in a permission mode: full (acts freely) or read-only (best-effort sandbox).

Why a panel beats a soloist. Spawning codex, claude, and a third model on the same question and having them cross-check — and then adversarially attack — each other's answers is the whole point. Independent models fail in different ways, so agreement across them is real signal, and disagreement flags exactly where to look harder.

Under the hood

How the coordination actually works.

You don't need any of this to use the platform — but if you're curious how one session drives a fleet of independent programs reliably, here's the machinery.

Coordination is files, not in-process messaging▸

The hub and workers never share memory or call each other directly. They talk through files in a per-worker state directory — simple, crash-safe, and inspectable. The core files:

  • inbox.md — the task the hub hands the worker.
  • outbox.jsonl — a stream of events the worker emits: ready, ack, progress, done, error, question.
  • status.json / pane.json — the worker's current state and which tmux pane it lives in.

The hub regains control between every step, reads the outbox, and decides what to do next. A worker that asks a question pauses; the hub answers (sometimes after asking you) and the worker resumes.

tmux is the only place subprocesses live▸

Every worker is a real interactive model TUI running in a tmux pane. That's what makes them attachable: you can tmux select-pane to watch any worker think and type, or jump in and take over. The pane border is colored by the worker's cluster so a screen full of crews stays legible. tmux is the single subprocess surface — nothing else shells out.

The wire protocol is frozen so models are drop-in▸

The event names, the JSON field names, the sentinel that marks the end of an instruction, and the state file layout are frozen — deliberately never renamed. That's what lets a new external model binary slot in as just another provider without touching the orchestration logic. Adding a provider is a config row plus a live test, by design.

Atomic writes, absolute paths, single-repo▸

State files are written atomically — to a temp file in the same directory, then renamed — so a crash never leaves a half-written status. All state paths are absolute and keyed to the repo, so runs in different repos never collide. The platform is intentionally single-repo: each run operates on one repository, and cross-repo work is the explicit job of /ap:bridge.

Zero-build install▸

The whole plugin ships as one committed bundle dispatched by subcommand, so installing it requires no build step — add the marketplace, install the plugin, and the /ap:* commands are available in your next session.

In one breath

What makes it worth the machinery.

  • One conductor, many specialists. A single session orchestrates a panel of different AI models instead of leaning on one.
  • Adversarial by default. Workers don't just answer — they attack each other's answers, so weak conclusions get caught before they reach you.
  • Everything is visible. Workers are real terminal panes you can attach to, and every run leaves forensics you can review later.
  • A clear path from question to PR. explore → design → implement takes an open question all the way to merged code, with on-ramps for quick fixes and real experiments.
